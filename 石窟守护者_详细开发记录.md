## 项目概览

- **类型**：2D 顶视角石窟文物保护巡检模拟游戏
- **核心机制**：4维属性（声誉/风险/证据/预算）+ 5任务×2步骤×3选择决策树 + 7种叙事绑定小游戏
- **地图**：100×70瓦片程序化敦煌暖褐色调，7房间+9走廊
- **视觉风格**：暗色调（#12110d背景）+ 金色高亮（#d7bd73）+ 石质UI面板
- **结局**：6种结局标签（石窟之殇 / 石窟守护者 / 吹哨人 / 档案专家 / 激进改革者 / 勉力维持者）
- **系统**：18成就 + 4挑战模式 + NG+遗产加成 + 3槽存档 + 属性转化 + 游客NPC生态

## 2026年6月16日 — 最大量开发日（25次修复/重构）

### 加载进度条 Bug 修复序列（17轮迭代，v5→v17）

这是整个项目中最顽固的Bug，经历了反复诊断和修复。

**v5: delay() 加入 double rAF 强制绘制**
- 问题：v4 setTimeout(150) 日志正常但视觉不更新
- 修复：setTimeout(150ms) → requestAnimationFrame → requestAnimationFrame → resolve
- 这是浏览器渲染领域公认的「强制绘制」模式

**v6: rAF 动画循环驱动进度条**
- v5方案导致游戏无法启动（rAF冲突）
- 重构为 targetProgress + currentProgress + animateProgress() rAF 循环
- 分离关注点：generate.ts管理纹理生成时序，main.tsx管理视觉动画

**v7: 页面刷新后自动启动游戏 → 进度条驱动 → TitleScreen**
- 根因：#root-loader 一直可见覆盖TitleScreen，但游戏需点击"开始游戏"才启动
- 修复：App.tsx 新增 'loading' 阶段 + 自动启动游戏 + game:ready 后显示标题界面

**v7同步发现三个Bug**
- Bug 1: 同步调用栈阻塞DOM渲染（最严重）— generateTextures() 中10次reportBootProgress() 全部在同一JS帧内同步执行
- Bug 2: CSS transition加剧感知差 — .bar-fill 有 transition: width 0.3s，与纹理生成实际耗时严重不匹配
- Bug 3: 重复15%报告 + 无异常处理

**v7修复：rAF驱动JS插值 + 分帧生成 + error handling**
- main.tsx: targetProgress/currentProgress 双变量 + animateProgress() rAF 循环
- generate.ts: generateTexturesAsync 分帧生成（6阶段，每阶段间 await rAF）
- BootScene: try/catch包裹 + 去重
- 进度流程: 5%→12%→15%→20%→30%→45%→60%→75%→85%→98%→100%

**v8: 进度条加载完后重复出现第二个进度条**
- 根因：App.tsx 在 game:ready 后将Phaser容器设为display:none → Canvas丢失渲染上下文 → Phaser内部重初始化 → BootScene再跑
- 修复：游戏容器始终可见 + loaderDismissed防重复标志

**v9: 进度条完成后黑屏，TitleScreen不出现**
- 根因：bus.on('game:ready') 与 window.addEventListener('boot:progress') 两种事件机制间的时序问题
- 修复：进度100%时派发 boot:complete 窗口事件，App.tsx 用 useEffect 监听

**v10: 进度条无动画 + 标题界面黑屏**
- 症状1：进度条始终停在3%然后直接跳标题画面
- 症状2：标题界面完全黑屏
- 修复：currentProgress 与初始测试对齐 + TitleScreen 外包裹 position:fixed; zIndex:9999

**v11: 真正根因 — React StrictMode 破坏了 autoCreateRef**
- 根因：React 18 StrictMode unmount+remount → autoCreateRef.useRef在unmount时被清理 → setTimeout被clearTimeout取消 → 游戏从未创建
- 修复：useRef → 模块级 let autoGameCreated = false（模块变量不受组件remount影响）
- rAF循环 → CSS transition（浏览器原生更可靠）

**v12: 点击开始游戏后出现"两个进度条"**
- 根因：旧游戏的 generateTexturesAsync 中的 await delay() 回调在 destroy() 后仍存活 → stale事件泄漏 → boot:reset把loaderDismissed重置 → 被新周期接收
- 修复：代际隔离机制 — bootGeneration 计数器 + event detail 携带 gen 字段 + isLive() 函数级abort
- 三层防护：generate.ts函数级abort → main.tsx事件级过滤 → clearTimeout定时器清理

**v13: Phaser画布进度条透出**
- 症状：v12修复后点击"开始游戏"第一个进度条加载完，第二个一闪而过
- 根因：BootScene在Phaser画布上绘制了自己的进度条，HTML loader半透明时透过可见
- 修复：移除BootScene画布进度条 + 新增loaderLocked三层防护 + isLive()异常日志

**v14: 点击开始游戏后GameScene一闪而过**
- 症状：旧游戏的GameScene画面一闪而过才出现加载条
- 根因：setPhase('playing')在setTimeout 100ms之前同步执行，TitleScreen消失后旧Phaser实例的GameScene露出
- 修复：destroyGame()先行 → setPhase('loading') → boot:reset → createGame()
- pendingGameStart标志区分初始加载和游戏重启

**v15: destroyGame后bus事件stale handler崩溃**
- 症状：`[EventBus] handler for "task:completed" threw: TypeError: Cannot read properties of null (reading 'add')`
- 根因：destroyGame()销毁旧Phaser实例，但setupBusListeners()注册的13个bus.on()监听器未被清理
- 修复：新增 busUnsubs[] 数组收集所有unsubscribe函数 + shutdown时统一取消订阅

**v16: 支线任务与主线任务同时活跃时的冲突**
- 症状：接取支线→再接主线→做完主线后显示"支线任务失败"
- 根因：handleTaskAdvance()只检查activeSideQuest是否为null，不检查activeTask
- 修复：增加 activeTask 检查，两者同时活跃时优先主线

**v17: TimingGame滑条窗口大小不固定导致无法点击**
- 根因：maxWidth: '100%' 将滑条缩放到非400px → 内部百分比定位的容器基准不固定
- 修复：flexShrink: 0 + minWidth: 500 确保滑条始终400px

### Bug修复: 转化面板返回后小地图消失
- 根因：phase从conversion切回playing时canvas DOM被销毁重建，GameScene缓存了旧canvas引用
- 修复：drawMiniMap()改为每帧动态查询document.getElementById('minimap-canvas')

## 核心系统架构概览

### 决策引擎
```
5任务 × 2步骤 × 3选择(专业/妥协/激进) = 30个独立选项
→ 每个选项绑定 minigame 类型+难度+属性delta+叙事文本
→ handleChoice → minigame/memory/timing/calibrate/wire/sequence/trace/puzzle
→ success/failure → handleStatsChange(deltas) → riskSystem计算
→ handleTaskAdvance → advanceTask → 推进步骤/触发故事事件
```

### 四维属性经济
```
声誉 40→0~100（S级90+门槛）
风险 40→0~100（≥80强制结束）
证据 10→0~100（溢出自动60→reputation 2:1）
预算 12→0~20（cap=20）
属性转化: 6种一次性操作（sell_data/report_fraud/emergency_repair/donate/cover_up/media_alert）
转化率区间: 1.5-1.67
```

### 结局判定
```
优先级递减:
石窟之殇(risk≥80强制) > 石窟守护者(reputation≥80+evidence≥60) > 
吹哨人(reputation≥60+evidence≥40) > 档案专家(evidence≥80) > 
激进改革者(reputation≥70+risk≥60) > 勉力维持者(默认)
16条因果回溯映射
```
